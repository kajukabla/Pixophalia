# Black Background Guardrails Research

Research into ensuring AI-generated pixel art game sprites always have solid black backgrounds for our flood-fill transparency pipeline.

## Our Pipeline

Our rendering pipeline uses flood-fill from image edges to detect exterior black pixels and treats them as transparent. This means:
- The background MUST be solid black (#000000)
- Any non-black background (checkerboard, white, colored, gradient) breaks the pipeline
- Anti-aliased edges blending into non-black backgrounds create artifacts

## 1. Prompting Guardrails

### General Best Practices

- **Be explicit and redundant**: Include background instructions at both the START and END of the prompt. Example: "On a solid black background, ... The entire background must be solid black (#000000) with no gradients, patterns, or other colors."
- **Specify what NOT to do**: "No transparency patterns, no checkerboard, no gradients, no colored backgrounds"
- **Use hex codes**: "#000000 black background" is more precise than just "black background"
- **Include rendering context**: "Pixel art sprite isolated on a solid black background, suitable for sprite sheet extraction"
- **Avoid the word "transparent"**: All models interpret this differently. Gemini renders a checkerboard pattern. GPT models may cut holes in the image. Always say "solid black" instead.

### Model-Specific Prompt Strategies

#### nano-banana-2 / Gemini

Gemini models cannot generate true transparency and produce flat RGB images regardless of prompt. The checkerboard pattern occurs because the model was trained on stock images that display transparency using checkerboard overlays.

**Recommended approach**: Do NOT ask for black backgrounds with Gemini. Instead:
1. Request a **bright green (#00FF00) background** (chroma key approach)
2. Add "with a 2-3px white outline around the character" to create a buffer zone
3. Post-process: use HSV-based color detection to identify and remove green pixels, then composite onto black

Why green works better than black:
- Green has zero overlap with typical sprite palettes (unlike black, which appears in outlines, shadows, etc.)
- HSV detection can target green precisely without touching sprite content
- The white outline acts as a clean buffer between sprite and background
- Chroma keying is a solved, battle-tested VFX technique

**Alternative**: Request a **solid magenta (#FF00FF)** background -- another classic chroma key color with low overlap to typical palettes.

#### Flux Models (flux-2-max, flux-2-pro, flux-pro-1.1)

These models generally follow background instructions well. Key tips from [Flux prompt guides](https://getimg.ai/blog/flux-1-prompt-guide-pro-tips-and-common-mistakes-to-avoid):
- Use descriptive, specific natural language
- Place background instruction early in prompt: "A pixel art knight sprite on a solid black (#000000) background..."
- Specify exact HEX colors for precise matching
- Include the keyword "color" or "hex" before the code for best results
- Avoid the word "transparent" -- it can cause fuzzy outputs
- Use 30-50 denoising steps for optimal detail retention
- Keep aspect ratio 1:1 (square) for sprite generation

#### Flux Kontext (kontext-max, kontext-pro)

Kontext models are primarily designed for image EDITING, not generation from scratch. They tend to interpret prompts as scene descriptions rather than isolated sprite requests.

**Strategies**:
- Be extremely explicit: "Generate ONLY the character sprite with absolutely nothing else. The ENTIRE background must be solid black #000000. No scenery, no landscape, no environment."
- Add constraints: "Do not generate any environment, ground, sky, or scene elements"
- Consider using Kontext only for image-to-image editing (e.g., taking a sprite generated by flux-2-pro and modifying it) rather than text-to-image generation
- If used for generation, add preservation instructions: "Keep the background as solid black #000000, do not add any scene elements"

#### GPT Image Models (gpt-image-1.5)

OpenAI's API has a dedicated `background` parameter with three options: `auto`, `transparent`, `opaque`.

**Strategies**:
- Set `background: "opaque"` in the API call (not "transparent" or "auto")
- In the prompt, explicitly state: "on a solid black (#000000) background"
- The `transparent` option works but outputs actual alpha transparency -- which would need compositing onto black for our pipeline
- If using transparent mode: export as PNG/WebP (JPEG doesn't support transparency), set quality to "medium" or "high"
- GPT models respond well to structured prompts: separate background, subject, and style instructions clearly

**Warning**: GPT image models sometimes generate white/light backgrounds even when asked for black. The `background: "opaque"` parameter does NOT guarantee black -- it just guarantees no transparency. You must ALSO specify black in the prompt text.

## 2. Post-Processing Guardrails

Even with perfect prompts, models will sometimes produce incorrect backgrounds. Post-processing is essential.

### Strategy A: Corner Sampling Validation + Reject/Retry

Sample pixels at the four corners and along edges. If they're not close to black, reject the image and retry with a modified prompt.

```
Pseudocode:
corners = [(0,0), (w-1,0), (0,h-1), (w-1,h-1)]
edge_samples = sample N pixels along each edge
for pixel in corners + edge_samples:
    if brightness(pixel) > threshold (e.g., 10):
        return FAIL
return PASS
```

This is fast, cheap, and catches the most common failures (white bg, colored bg, checkerboard).

### Strategy B: Composite Onto Black Canvas

For images with actual transparency (alpha channel from GPT transparent mode):
1. Create a new black (#000000) image of the same dimensions
2. Paste/composite the generated image onto it using alpha compositing
3. Export as opaque PNG

This handles the case where a model generates actual transparency instead of solid black.

### Strategy C: Chroma Key Post-Processing (for Gemini)

For images generated with a green/magenta background:
1. Convert image to HSV color space
2. Create a mask for pixels matching the chroma key color (within tolerance)
3. Replace masked pixels with black (#000000)
4. Optional: erode the mask slightly to clean up edge artifacts

```
Pseudocode:
hsv = convert_to_hsv(image)
green_mask = (hue in [35-85]) AND (saturation > 100) AND (value > 100)
image[green_mask] = (0, 0, 0)
```

### Strategy D: Edge Flood-Fill Background Replacement

If the background is an unknown color (not black, not a known chroma key):
1. Sample the dominant color from corners/edges
2. Flood-fill from all four corners using that color as the seed (with tolerance)
3. Replace all flood-filled pixels with black
4. This preserves the sprite content while fixing the background

This is essentially running our transparency pipeline in reverse -- find what WOULD be transparent, and make it black.

### Strategy E: Brightness-Based Background Replacement

For cases where the background is lighter than the sprite:
1. Sample corner pixels to determine background brightness
2. If background is light (white/gray), invert detection: flood-fill from edges where brightness > threshold
3. Replace those pixels with black

### Strategy F: Image-to-Image Correction Pass

Use a model like Flux Kontext in image-to-image mode:
1. Take the failed generation as input
2. Prompt: "Change the background to solid black #000000. Keep the character sprite exactly the same."
3. This leverages the model's editing capabilities to fix just the background

Cost: extra API call. But useful as a fallback when programmatic approaches fail (e.g., complex backgrounds that flood-fill can't cleanly separate).

## 3. Model-Specific Strategies Summary

### Tier 1: Most Reliable (use for primary generation)
- **flux-2-pro**: Best overall reliability for black backgrounds. Good prompt adherence, consistent results.
- **flux-2-max**: Higher quality than pro, equally reliable for backgrounds. Use for hero/key assets.
- **flux-pro-1.1**: Reliable, good for batch generation.

### Tier 2: Reliable with Caveats
- **gpt-image-1.5**: Use `background: "opaque"` API parameter + explicit black prompt. Validate output. Has the advantage of a dedicated background parameter, but still needs prompt reinforcement.

### Tier 3: Requires Special Handling
- **flux-kontext-max / flux-kontext-pro**: Best used for image-to-image editing, not text-to-image sprite generation. If used for generation, needs aggressive prompt constraints and high retry rate. Consider using ONLY as a second pass to edit sprites generated by Tier 1 models.
- **nano-banana-2 (Gemini)**: Cannot reliably produce black backgrounds. Use the green-screen chroma key approach instead. Always requires post-processing. Has unique strengths (style, character design) that make it worth using despite the background issues.

## 4. Validation Function Design

### Multi-Level Validation Pipeline

```
Level 1: FAST - Corner Check (< 1ms)
  Sample 4 corner pixels + 4 edge midpoints
  All must be within threshold of black (R,G,B each < 10)
  PASS -> accept image
  FAIL -> proceed to Level 2

Level 2: MEDIUM - Edge Scan (< 10ms)
  Sample every Nth pixel along all 4 edges
  Calculate percentage that are "black" (brightness < threshold)
  If > 95% black -> PASS (minor artifacts, acceptable)
  If > 50% black -> WARN (possible partial background issue)
  If < 50% black -> FAIL (wrong background color entirely)

Level 3: DEEP - Background Region Analysis (< 100ms)
  Flood-fill from corner to find connected background region
  Calculate total area of background region
  Check if all pixels in region are near-black
  Detect specific failure patterns:
    - Checkerboard: alternating light/dark pixels in grid pattern
    - Gradient: smooth brightness transition across background
    - Colored background: non-black hue detected in background region
    - Scene background: high color variance in background region
  Return detailed diagnostic for retry prompt adjustment
```

### Checkerboard Detection (for Gemini outputs)

```
Pseudocode:
# Sample a patch from the corner (e.g., 8x8 pixels)
patch = image[0:8, 0:8]
# Check for alternating pattern
even_pixels = patch[0::2, 0::2]  # pixels at even row, even col
odd_pixels = patch[1::2, 1::2]   # pixels at odd row, odd col
cross_pixels = patch[0::2, 1::2] # pixels at even row, odd col

# If even/odd positions have very different brightness, it's a checkerboard
mean_diff = abs(mean(even_pixels) - mean(cross_pixels))
if mean_diff > 50:
    return CHECKERBOARD_DETECTED
```

### Retry Strategy on Validation Failure

When validation fails, the retry should adjust based on the failure type:

| Failure Type | Retry Strategy |
|-------------|---------------|
| White/light background | Add "NOT white, NOT light" to prompt. Emphasize "#000000 solid black" |
| Colored/scene background | Add "No scenery, no environment, ONLY the sprite on black" |
| Checkerboard (Gemini) | Switch to green-screen prompt + chroma key post-processing |
| Gradient background | Add "perfectly uniform black, no gradients, no lighting effects on background" |
| Partial background (edges OK, center not) | Likely a scene issue -- add "isolated sprite, nothing behind the character" |

### Maximum Retry Logic

```
max_retries = 3
for attempt in range(max_retries):
    image = generate(prompt)
    result = validate(image)
    if result.passed:
        return image
    prompt = adjust_prompt(prompt, result.failure_type)

# If all retries fail, apply post-processing fix
image = force_black_background(image)  # composite/flood-fill approach
final_result = validate(image)
if not final_result.passed:
    flag_for_manual_review(image)
```

## 5. Recommended Implementation Order

1. **Implement validation function** (Level 1 + Level 2) -- catches obvious failures immediately
2. **Add "composite onto black" post-processing** -- fixes transparency/alpha issues from GPT models automatically
3. **Add chroma key pipeline for Gemini** -- switch nano-banana-2 prompts to green screen + HSV removal
4. **Add retry logic with prompt adjustment** -- automatically retry on validation failure
5. **Add flood-fill background replacement** -- fallback for unknown/complex background colors
6. **Consider image-to-image correction** -- expensive but handles edge cases no other method catches
